O que vamos construir?
O panorama completo do projeto

Vamos criar uma API REST para um sistema acadêmico que gerencia alunos, professores, disciplinas e matrículas. No final, o sistema vai:

Analogia: A API é um garçom

Pense na API como um garçom de restaurante. O cliente (frontend/Insomnia) faz o pedido, o garçom (Controller) anota, leva para a cozinha (Service), que pega os ingredientes na despensa (Repository/Banco). O prato pronto (JSON) volta pelo garçom.

Arquitetura em Camadas
Cada camada tem uma responsabilidade
🤔 Por que dividir em camadas?

Imagine uma fábrica: cada setor tem uma função. O RH não fabrica peças, e a linha de produção não cuida das finanças. No software é igual — separação de responsabilidades torna o código organizado, testável e fácil de manter.

O que é o Model? É a classe Java que representa uma tabela do banco de dados. Cada atributo da classe vira uma coluna. Por exemplo, a classe AlunoModel com os atributos nome e email gera automaticamente uma tabela aluno com as colunas nome e email. É por isso que chamamos de mapeamento objeto-relacional (ORM) — o Hibernate traduz seus objetos Java em estrutura de banco!
💡
O Spring Boot gera o SQL automaticamente! Você trabalha com objetos Java e o Hibernate traduz tudo para SQL. Isso se chama ORM (Object-Relational Mapping).

💉 Injeção de Dependência — @Autowired

O Spring "injeta" automaticamente as dependências entre as camadas. É como uma seringa que conecta tudo pra você!

🤔 O que é @Autowired?

Quando o Controller precisa usar o Service, você não faz new Service() manualmente. O Spring injeta automaticamente a dependência — como uma seringa que entrega o componente certo no lugar certo. Isso é o padrão de Inversão de Controle (IoC).

🗄️ Repository @Repository
@Autowired injeta Repository no Service
⚙️ Service @Service
@Autowired injeta Service no Controller
🎮 Controller @RestController
💉
Na prática: Quando você declara @Autowired private AlunoService alunoService; dentro do Controller, o Spring automaticamente cria e injeta uma instância do Service. Você nunca precisa fazer new AlunoService() — o Spring cuida disso!

🤔 O que é um Model?

O Model é uma classe Java que representa uma tabela no banco de dados. Cada atributo da classe vira uma coluna na tabela. Cada objeto que criamos vira uma linha na tabela.

Analogia: Ficha de Cadastro

O Model é como uma ficha de cadastro em branco. A ficha define os campos (nome, CPF, email). Quando você preenche uma ficha, é como criar um objeto. Quando guarda no arquivo, é como salvar no banco!

package br.com.alunoonline.api.model;

import jakarta.persistence.*;
import lombok.*;

@NoArgsConstructor  // Gera construtor vazio@NoArgsConstructorO Lombok gera automaticamente um construtor sem parâmetros: public Aluno() {}. O JPA exige esse construtor para criar objetos a partir do banco.
@AllArgsConstructor // Gera construtor com todos os campos@AllArgsConstructorGera um construtor com TODOS os atributos: public Aluno(Long id, String nome, String cpf, String email) {}
@Data                // Gera getters, setters, toString, equals@DataAnotação do Lombok que gera automaticamente: getters, setters, toString(), equals(), hashCode(). Sem ela, você teria que escrever ~60 linhas de código!
@Table(name = "aluno")   // Nome da tabela no banco@Table(name = "aluno")Define o nome da tabela no PostgreSQL. Sem essa anotação, o Hibernate usaria o nome da classe como nome da tabela.
@Entity              // "Esta classe é uma tabela!"@EntityDiz ao JPA: "esta classe Java representa uma tabela no banco de dados". É a anotação mais importante do Model!
public class Aluno {

    @Id                          // "Este campo é a chave primária"@IdMarca este campo como a PRIMARY KEY da tabela. Todo @Entity precisa ter um @Id.
    @GeneratedValue(strategy = GenerationType.IDENTITY)@GeneratedValue(strategy = IDENTITY)O banco gera o ID automaticamente (auto-increment). Você NÃO precisa definir o ID ao criar um aluno — o PostgreSQL cuida disso!
    private Long id;

    private String nome;

    private String cpf;

    private String email;
}
🏷️ O que cada anotação faz? (Clique para expandir) ▼
@Entity
"Esta classe é uma tabela no banco!" — Sem isso, o JPA ignora a classe.
@Table
Define o nome da tabela. Sem ela, usa o nome da classe.
@Id
Marca o campo como PRIMARY KEY. Obrigatório em toda @Entity.
@GeneratedValue
O banco gera o valor automaticamente (auto-increment).
@Data
Lombok: gera getters, setters, toString, equals, hashCode.
@NoArgsConstructor
Lombok: gera construtor vazio (exigido pelo JPA).

🗄️ O que é um Repository?

O Repository é uma interface (não uma classe!) que herda de JpaRepository. Ao estender ela, você ganha de graça vários métodos para operar no banco — sem escrever nenhum SQL!

Analogia: Funcionário do Arquivo

O Repository é como um funcionário do arquivo morto. Você pede: "guarde esta ficha", "me dê a ficha do ID 5", "me dê todas as fichas". Ele faz tudo sem você saber como o arquivo está organizado internamente.

import org.springframework.stereotype.Repository;

@Repository@RepositoryMarca esta interface como um componente de acesso a dados do Spring. Ajuda no tratamento de exceções do banco.
public interface AlunoRepository extends JpaRepository<Aluno, Long> {
}  // Sim, está vazio! E já funciona! 🪄
🪄
JpaRepository<Aluno, Long> significa: "este repositório gerencia a entidade Aluno e sua chave primária é do tipo Long". Ao herdar, você ganha de graça:
.save(aluno)
INSERT ou UPDATE — salva no banco
.findAll()
SELECT * — retorna todos
.findById(id)
SELECT WHERE id = ? — busca por ID
.deleteById(id)
DELETE WHERE id = ? — deleta por ID

⚙️ O que é um Service?

O Service é a camada de regras de negócio. É aqui que a lógica acontece: validações, cálculos, decisões. O Controller nunca acessa o Repository diretamente — sempre passa pelo Service.

Analogia: O gerente

O Service é como um gerente. O atendente (Controller) recebe o pedido do cliente, mas quem decide o que fazer é o gerente (Service). O gerente delega o trabalho braçal para o estagiário (Repository).

package br.com.alunoonline.api.service;

import br.com.alunoonline.api.model.Aluno;
import br.com.alunoonline.api.repository.AlunoRepository;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.http.HttpStatus;
import org.springframework.stereotype.Service;
import org.springframework.web.server.ResponseStatusException;
import java.util.List;
import java.util.Optional;

@Service  // "Esta classe é um Service do Spring!"@ServiceMarca a classe como um componente de serviço do Spring. O Spring cria uma instância automaticamente e gerencia o ciclo de vida dela.
public class AlunoService {

    @Autowired  // Injeção de dependência automática!@AutowiredO Spring injeta automaticamente uma instância do AlunoRepository aqui. Você NÃO precisa fazer new AlunoRepository() — o Spring cuida disso!
    AlunoRepository alunoRepository;

    // CREATE — salvar novo aluno
    public void criarAluno(Aluno aluno) {
        alunoRepository.save(aluno);  // INSERT INTO aluno ...
    }

    // READ — listar todos
    public List<Aluno> listarTodosAlunos() {
        return alunoRepository.findAll();  // SELECT * FROM aluno
    }

    // READ — buscar por ID
    public Optional<Aluno> buscarAlunoPorId(Long id) {
        return alunoRepository.findById(id);  // SELECT * FROM aluno WHERE id = ?
    }

    // UPDATE — atualizar por ID
    public void atualizarAlunoPorId(Long id, Aluno aluno) {
        Optional<Aluno> alunoDoBanco = buscarAlunoPorId(id);

        if (alunoDoBanco.isEmpty()) {
            throw new ResponseStatusException(
                HttpStatus.NOT_FOUND, "Aluno não encontrado");
        }

        Aluno alunoParaEditar = alunoDoBanco.get();
        alunoParaEditar.setNome(aluno.getNome());
        alunoParaEditar.setCpf(aluno.getCpf());
        alunoParaEditar.setEmail(aluno.getEmail());

        alunoRepository.save(alunoParaEditar);  // UPDATE aluno SET ...
    }
}
🔍 O que é Optional e por que usamos? ▼

Optional<Aluno> é como uma caixa que pode estar vazia ou conter um aluno.

Quando buscamos por ID, o aluno pode não existir. O Optional nos obriga a tratar esse caso:

  • .isEmpty() → a caixa está vazia? (aluno não existe)
  • .get() → pega o aluno de dentro da caixa
  • .isPresent() → tem algo na caixa?

Isso evita o famoso NullPointerException! 💀

🔍 O que é @Autowired? ▼

Injeção de Dependência — O Spring cria os objetos pra você!

Sem @Autowired, você teria que fazer: AlunoRepository repo = new AlunoRepository()

Com @Autowired, o Spring faz isso automaticamente quando a aplicação inicia. É como ter um assistente que prepara tudo antes de você começar a trabalhar.

package br.com.alunoonline.api.controller;

import br.com.alunoonline.api.model.Aluno;
import br.com.alunoonline.api.service.AlunoService;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.http.HttpStatus;
import org.springframework.web.bind.annotation.*;
import java.util.List;
import java.util.Optional;

@RestController          // "Esta classe é um controller REST!"@RestControllerCombina @Controller + @ResponseBody. Diz ao Spring que esta classe recebe requisições HTTP e retorna JSON automaticamente.
@RequestMapping("/alunos")  // URL base: /alunos@RequestMapping("/alunos")Define a URL base. Todos os endpoints dessa classe começam com /alunos.
public class AlunoController {

    @Autowired
    AlunoService alunoService;

    @PostMapping                     // POST /alunos@PostMappingEste método responde a requisições HTTP POST. Usado para CRIAR recursos.
    @ResponseStatus(HttpStatus.CREATED)  // Retorna 201@ResponseStatus(CREATED)Define que a resposta HTTP terá o status 201 (Created) em vez do padrão 200 (OK).
    public void criarAluno(@RequestBody Aluno aluno) {
        alunoService.criarAluno(aluno);
    }

    @GetMapping                      // GET /alunos@GetMappingResponde a requisições HTTP GET. Usado para BUSCAR/LISTAR recursos.
    @ResponseStatus(HttpStatus.OK)
    public List<Aluno> listarTodosAlunos() {
        return alunoService.listarTodosAlunos();
    }

    @GetMapping("/{id}")              // GET /alunos/1@GetMapping("/{id}")O {id} é um parâmetro de caminho (path variable). Se a URL for /alunos/5, o id será 5.
    @ResponseStatus(HttpStatus.OK)
    public Optional<Aluno> buscarAlunoPorId(@PathVariable Long id@PathVariableCaptura o valor da URL. Em /alunos/5, o parâmetro id recebe o valor 5.) {
        return alunoService.buscarAlunoPorId(id);
    }

    @PutMapping("/{id}")              // PUT /alunos/1@PutMappingResponde a requisições HTTP PUT. Usado para ATUALIZAR recursos completamente.
    @ResponseStatus(HttpStatus.NO_CONTENT)  // 204
    public void atualizarAlunoPorId(@PathVariable Long id,
                                    @RequestBody Aluno aluno@RequestBodyConverte o JSON do corpo da requisição para um objeto Java automaticamente. O Spring faz a "mágica" de transformar JSON → Objeto.) {
        alunoService.atualizarAlunoPorId(id, aluno);
    }
}

Novidade: @ManyToOne — relacionamento com Professor!
🔗 O que é @ManyToOne?

Muitas disciplinas para um professor. Um professor pode lecionar várias disciplinas, mas cada disciplina tem apenas um professor. Isso é um relacionamento Muitos-para-Um (N:1).

Analogia: Filhos e Pais

Vários filhos (disciplinas) podem ter o mesmo pai (professor), mas cada filho tem apenas um pai. A disciplina "aponta" para o professor dela — como se tivesse um campo "quem é meu professor?".

package br.com.alunoonline.api.model;

import jakarta.persistence.*;
import lombok.*;

@NoArgsConstructor
@AllArgsConstructor
@Data
@Table(name = "disciplina")
@Entity
public class Disciplina {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String nome;

    private Integer cargaHoraria;

    @ManyToOne  // "Muitas disciplinas → um professor"@ManyToOneDefine o relacionamento N:1 (Many-to-One). Muitas disciplinas podem apontar para um mesmo professor. No banco, isso cria uma FOREIGN KEY.
    @JoinColumn(name = "professor_id")  // Nome da FK no banco@JoinColumn(name = "professor_id")Define o nome da coluna de FOREIGN KEY na tabela disciplina. Essa coluna guarda o ID do professor associado.
    private Professor professor;
}

package br.com.alunoonline.api.repository;

import br.com.alunoonline.api.model.Disciplina;
import org.springframework.data.jpa.repository.JpaRepository;
import org.springframework.stereotype.Repository;
import java.util.List;

@Repository
public interface DisciplinaRepository extends JpaRepository<Disciplina, Long> {

    // O Spring gera o SQL: SELECT * FROM disciplina WHERE professor_id = ?
    List<Disciplina> findByProfessorId(Long professorId);
}
Como funciona a "magia"?

O Spring Data lê o nome do método e desmonta assim:
find → SELECT
By → WHERE
ProfessorId → professor_id = ?
Resultado: SELECT * FROM disciplina WHERE professor_id = ?

O que é um Enum?

Enum é uma lista fixa de opções. Em vez de usar Strings como "APROVADO" (que pode ser digitada errada como "aprovado" ou "APROVDO"), o Enum garante que só existem 4 valores possíveis. É como um menu fechado — só pode escolher o que está lá!

package br.com.alunoonline.api.service;

import br.com.alunoonline.api.enums.MatriculaAlunoStatusEnum;
import br.com.alunoonline.api.model.MatriculaAluno;
import br.com.alunoonline.api.repository.MatriculaAlunoRepository;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.http.HttpStatus;
import org.springframework.stereotype.Service;
import org.springframework.web.server.ResponseStatusException;

@Service
public class MatriculaAlunoService {

    private static final Double MEDIA_PARA_APROVACAO = 7.0;

    @Autowired
    MatriculaAlunoRepository matriculaAlunoRepository;

    // ═══════════════ CRIAR MATRÍCULA ═══════════════
    public void criarMatricula(MatriculaAluno matriculaAluno) {
        matriculaAluno.setStatus(
            MatriculaAlunoStatusEnum.MATRICULADO);  // Status automático!
        matriculaAlunoRepository.save(matriculaAluno);
    }

    // ═══════════════ TRANCAR MATRÍCULA ═══════════════
    public void trancarMatricula(Long id) {
        MatriculaAluno matricula =
            matriculaAlunoRepository.findById(id)
                .orElseThrow(() -> new ResponseStatusException(
                    HttpStatus.NOT_FOUND,
                    "Matricula não encontrada"));

        // Só pode trancar se estiver MATRICULADO!
        if (matricula.getStatus()
                .equals(MatriculaAlunoStatusEnum.MATRICULADO)) {
            matricula.setStatus(
                MatriculaAlunoStatusEnum.TRANCADO);
            matriculaAlunoRepository.save(matricula);
        } else {
            throw new ResponseStatusException(
                HttpStatus.BAD_REQUEST,
                "Só é possível trancar com status MATRICULADO");
        }
    }

    // ═══════════════ ATUALIZAR NOTAS ═══════════════
    public void atualizarNotas(Long id,
                               AtualizarNotasRequestDTO dto) {
        MatriculaAluno matricula = buscarOuLancarExcecao(id);

        if (dto.getNota1() != null)
            matricula.setNota1(dto.getNota1());
        if (dto.getNota2() != null)
            matricula.setNota2(dto.getNota2());

        // Se ambas as notas existem, calcula a média!
        if (matricula.getNota1() != null
            && matricula.getNota2() != null) {
            Double media = (matricula.getNota1()
                          + matricula.getNota2()) / 2;
            matricula.setStatus(
                media >= MEDIA_PARA_APROVACAO
                    ? MatriculaAlunoStatusEnum.APROVADO
                    : MatriculaAlunoStatusEnum.REPROVADO);
        }

        matriculaAlunoRepository.save(matricula);
    }
}
🧮
Regra de aprovação: Se nota1 e nota2 existem → calcula média = (nota1 + nota2) / 2. Se média ≥ 7.0 → APROVADO. Se média < 7.0 → REPROVADO.

DTOs — Data Transfer Objects

Por que não usamos o Model diretamente? Entenda agora!

5.1
O que é DTO e por que usar?
Agora que você já sabe Model, hora de evoluir!
📦 DTO = Data Transfer Object

Um DTO é uma classe simples que serve apenas para transportar dados entre camadas. Diferente do Model (que representa a tabela), o DTO representa o que o cliente envia ou recebe.

Por que não usar o Model direto?

Imagine que a tabela aluno tem 20 campos, mas para atualizar as notas você só precisa de 2 campos (nota1 e nota2). Faz sentido obrigar o cliente a enviar todos os 20? Não! O DTO permite enviar só o necessário. É como preencher apenas os campos relevantes de um formulário, em vez de um formulário gigante.

💡
Quando começamos a usar DTOs neste projeto? Só aqui no Capítulo 5! Primeiro aprendemos com o Model direto para entender bem, e agora evoluímos para DTOs. É mais fácil assim!
